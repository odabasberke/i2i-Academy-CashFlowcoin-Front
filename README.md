# CryptoPal

A simulated crypto trading platform with AI-powered portfolio insights —
a full-stack portfolio project. Register, get a random demo balance,
trade BTC/ETH against live simulated prices, and ask an AI (Google
Gemini) questions about your own portfolio.

**Not real money, not a real exchange, not financial advice** — every
price is generated by an in-process simulator (see `market` below), and
the AI's answers are explicitly instructed to stick to your actual
account data.

## Stack

| Layer      | Technology                                                        |
|------------|--------------------------------------------------------------------|
| Backend    | Spring Boot 4.1.0 (Spring Framework 7), Java 25                    |
| Database   | PostgreSQL 18.4                                                    |
| Cache      | Redis 7 (`redis:7-alpine`)                                         |
| AI         | Google Gemini (`gemini-3.5-flash` by default) via `RestClient`     |
| Auth       | Stateless JWT (JJWT), BCrypt, Redis-backed session revocation      |
| Frontend   | React 19 + Vite, React Router, Axios, Tailwind CSS v4               |

## Architecture at a glance

The backend is a **modular monolith** — one deployable, six packages,
each owning a clear slice of the domain and talking to the others only
through narrow, explicit APIs (never reaching into another package's
repositories directly):

```
com.cryptopal
├── auth        registration/login, JWT issuance & validation, the request-auth filter
├── market      the simulated price feed, Redis price cache, cache-sync worker
├── trading     wallets, order execution (buy/sell), portfolio queries
├── ai          Gemini integration, portfolio-context prompt building
├── config      datasource/Redis/security/scheduling/async wiring
└── exception   custom exceptions + one global @RestControllerAdvice
```

A few decisions worth knowing about if you're reading the code:

- **Prices never touch Postgres on the read path.** A background thread
  (`market.SimulatedTickerEngine`) generates a random walk every 15s; a
  separate scheduled worker (`market.MarketCacheSyncWorker`) writes the
  latest tick to Redis (ephemeral, TTL'd) every 15s and folds it into a
  real OHLC candle in `price_trends` every 60s. Every price read — the
  public `/api/market/prices` endpoint, order execution, AI context — 
  reads Redis directly.
- **Every trade is one `@Transactional` boundary** with pessimistic row
  locks (`SELECT ... FOR UPDATE`) on the wallets involved, fiat wallet
  locked before the asset wallet in both directions — this makes
  concurrent orders from the same user deadlock-free and prevents
  lost-update races on balances.
- **JWT + Redis, together:** a JWT proves who you are without a DB
  round-trip; a matching `session:<jti>` key in Redis is what actually
  lets logout revoke a token before its natural 60-minute expiry.
- **The AI call never blocks a request thread.** `ai.AiInsightService`
  runs on a dedicated bounded thread pool (`@Async`, not WebFlux) and the
  controller returns a `CompletableFuture` — Spring MVC releases the
  thread immediately and completes the response once Gemini answers (or
  times out).
- **`auth` and `trading` don't depend on each other's types.** The JWT
  filter puts the authenticated user's numeric ID in
  `Authentication.getDetails()`; `trading` and `ai` each read it via their
  own tiny (deliberately duplicated, not shared) `AuthenticatedUserId`
  helper. The two packages agree on a convention, not on a class.

## Getting started

### Prerequisites
- Docker + Docker Compose
- JDK 25 (or 21/17 — see `pom.xml`, the code only needs 17 as a hard
  floor)
- Node.js 20+ and npm
- A [Google Gemini API key](https://aistudio.google.com/apikey)

### 1. Start infrastructure

```bash
cp .env.example .env    # fill in POSTGRES_PASSWORD, JWT_SECRET, GEMINI_API_KEY
docker compose up -d    # Postgres + Redis; db/init.sql applies on first run only
```

### 2. Run the backend

```bash
mvn spring-boot:run
# or run com.cryptopal.CryptoPalApplication from your IDE
```

The app reads the same `.env` values as environment variables — either
export them in your shell, use an IDE run-configuration with an env
file, or a tool like `direnv`. **On Windows**, Spring Boot doesn't load
`.env` on its own the way `docker compose` does — run `.\run-backend.ps1`
from PowerShell instead, which loads `.env` into the session and then
starts the app.

Once it's up:
- API: http://localhost:8080/api
- Swagger UI: http://localhost:8080/swagger-ui.html

### 3. Run the frontend

```bash
cd frontend
npm install
npm run dev
```

Open http://localhost:5173. The dev server proxies `/api/**` to
`localhost:8080` (see `frontend/vite.config.js`), so there's no CORS
setup to do.

### 4. Try it out
1. Register a new account — you'll get a random $100–$1000 demo USD
   balance immediately.
2. Watch the live ticker at the top and the market table update every
   ~5s.
3. Click **Trade** on BTC or ETH, buy some, then sell part of it back.
4. Check **Recent activity** and your updated balances.
5. Ask the AI panel something like *"What's my portfolio worth right
   now?"*

## API reference

| Method | Path                  | Auth | Description                                      |
|--------|-----------------------|------|---------------------------------------------------|
| POST   | `/api/auth/register`  | —    | Create an account (+ starter wallet)              |
| POST   | `/api/auth/login`     | —    | Get a JWT                                         |
| GET    | `/api/market/prices`  | —    | Live prices for every supported symbol (Redis)    |
| GET    | `/api/market/prices/{symbol}` | — | Live price for one symbol                     |
| POST   | `/api/trading/order`  | JWT  | Execute a market BUY/SELL                         |
| GET    | `/api/portfolio`      | JWT  | Wallet balances + last 50 transactions            |
| POST   | `/api/ai/query`       | JWT  | Ask Gemini a question about your own portfolio    |

Full request/response shapes are in Swagger UI once the backend is
running.

## Testing

```bash
mvn test
```

Unit tests use JUnit 5 + Mockito + AssertJ throughout, favoring
constructor injection specifically so services can be tested without a
Spring context. One known gap: `ai.GeminiClient` isn't unit-tested
directly — mocking `RestClient`'s fluent chain gets fragile fast, so
`AiInsightService` is tested against a mocked `GeminiClient` instead, and
`GeminiClient` itself relies on careful code review.

## Known simplifications

Built as a portfolio project, not a production exchange — a few
deliberate corners were cut and are called out in code comments where
they live:

- **1 USDT ≈ 1 USD.** Simulated prices are quoted against USDT; wallets
  are quoted in USD. Treated as pegged 1:1 rather than adding real FX
  conversion.
- **No trading fees.** Every logged trade has `fee = 0`.
- **One ledger row per trade**, tied to the crypto wallet, not a
  double-entry pair — matches the `transaction_logs` schema's single
  `wallet_id` column.
- **Prices are simulated**, not sourced from a real exchange — see
  `market.SimulatedTickerEngine`'s random walk.
- **No conversation memory in AI chat** — each question is answered
  independently with freshly-gathered context; the frontend keeps chat
  history for display only.

    │   ├── pages/             # Login, Register, Dashboard
